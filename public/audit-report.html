🔴 Critical Findings

Must fix before next significant user-facing update.

Critical C1 — SubscriptionPage silently drops props: onGoHome and onBackToProfile are never used

What it is

SubscriptionPage is declared with no props interface (export function SubscriptionPage()), but App.tsx passes onGoHome and onBackToProfile to it. Both are silently ignored. The onBackToProfile callback is specifically designed to let a user navigate back to the survey at step 5 — that feature is completely broken.

Why it matters

TypeScript would flag this as a type error. The "back to profile" flow (used in the /survey?initialStep=5 path) does not work. Any user who wants to revise their survey answers before subscribing cannot do so.

How to fix

Add a props interface to SubscriptionPage: interface Props { onGoHome: () => void; onBackToProfile: () => void; } and wire them into the UI (e.g. a back button in the header).

Critical C2 — TPI state shown to premium users is hardcoded/fabricated, not real data

What it is

In Dashboard.tsx, the live TPI panel shown to premium users displays TpiPill value={visibleSignal?.analysis?.includes('Positive') ? 'Positive' : 'Neutral'} (line 477) for Medium Term, and always value="Neutral" (line 493) for the Value Indicator. Additionally, buildHistory() injects tpiMedium: 'Positive', tpiLong: 'Neutral' hardcoded for any live Supabase signal (lines 273–274).

Why it matters

Premium users are paying $19.99/month specifically for TPI state access. Showing them fabricated values (TPI inferred from whether the word "Positive" appears anywhere in analysis text) is both inaccurate and a product integrity issue. The Value Indicator is permanently "Neutral" for all users at all times.

How to fix

Add tpi_medium and tpi_long columns to the crypto_signals table. Store real values when inserting a signal. Read these fields directly in the Dashboard instead of inferring them from analysis text.

Critical C3 — SurveyFunnel ignores initialStep prop — TypeScript type error, retake flow broken

What it is

App.tsx passes initialStep={(location.state as { initialStep?: number })?.initialStep} to SurveyFunnel, but SurveyFunnelProps only declares onComplete and onBack. The initialStep prop is not in the interface and is never used inside the component. The component always starts at step 1 (or restores from sessionStorage).

Why it matters

The "back to profile" flow from the subscription page is supposed to drop the user at step 5 (the results screen). This is completely non-functional. TypeScript strict mode would catch this.

How to fix

Add initialStep?: number to SurveyFunnelProps and use it in the useState initializer: useState(parsedSurveyState?.step || initialStep || 1).

Critical C4 — alert() used for production errors in SubscriptionPage and SurveyFunnel

What it is

SubscriptionPage.tsx:134 uses alert(error.message) when checkout fails. SurveyFunnel.tsx:292 uses alert() when survey save fails.

Why it matters

alert() blocks the UI thread, looks completely out of place on a dark-themed premium fintech product, and can't be styled or dismissed gracefully. It's jarring on mobile.

How to fix

Replace with inline error state (already done correctly in every other form: AuthModal, AccountPage, ForgotPassword). Add an error state to both components and render an error banner like the pattern used elsewhere.

🟡 Warnings

Should be fixed — impact user experience, trust, or code maintainability.

Warning W1 — console.error() calls left in production code across 5 files

What it is

Found in: AuthContext.tsx:42, App.tsx:136, Dashboard.tsx:192, AccountPage.tsx:64, SubscriptionPage.tsx:56. These are real errors being silently logged to the browser console, visible to technically-minded users.

Why it matters

Leaks implementation details (table names, query structure). On most hosting platforms, console errors are ignored — if a Supabase query fails silently, the user sees nothing and support has no record of it.

How to fix

Route errors to a proper logging service (e.g. Sentry free tier) or at minimum surface them as UI error states rather than silent console logs.

Warning W2 — DELAY_MS defined twice in Dashboard.tsx (dead duplicate constant)

What it is

Dashboard.tsx:229 defines const DELAY_MS = 7 * 24 * 60 * 60 * 1000 and line 257 defines the identical const DELAY_MS_CONST = 7 * 24 * 60 * 60 * 1000. The first one is unused.

Why it matters

If someone changes one but not the other, the delay logic becomes inconsistent. The unused constant adds confusion.

How to fix

Remove the unused DELAY_MS and keep only DELAY_MS_CONST (or rename it to DELAY_MS).

Warning W3 — Dead blur code in Dashboard: isLatestBlurred is always false but conditionals remain

What it is

It's clear the blur approach was replaced by the "hide entirely" approach — but the dead conditionals were left in.

Why it matters

Adds ~60 lines of dead JSX, confusing anyone maintaining the code. The comment "No blur — new signals are hidden entirely" makes clear this is intentional — but the code wasn't cleaned up.

Remove the isLatestBlurred variable and all branches that depend on it. Keep only the non-blurred rendering paths.

Warning W4 — signal: any type in Dashboard table rendering

What it is

Dashboard.tsx:674: displayHistory.map((signal: any, i: number) — same on the mobile cards. The signal objects have a known shape (from the hardcoded arrays) but are typed as any in the render loop.

Why it matters

Suppresses TypeScript checking for the entire render path. Typos in property names (e.g. signal.prce) would be silently undefined instead of caught at compile time.

How to fix

Define a HistoryEntry type matching the shape of computeEarnings() output and use it in place of any.

Warning W5 — SubscriptionPage: no navigation header, no back button

What it is

The subscription page has no header, no logo, and no back/exit navigation. A user who arrives here mid-funnel has no obvious path except the browser back button.

Why it matters

Removes context for who the product is from (no logo). On mobile, many users disable the browser navbar. This increases bounce rate and reduces trust.

How to fix

Add a minimal sticky header with the Tradinsight logo and a back button (using the onGoHome prop once C1 is fixed).

Warning W6 — buildHistory() called on every render — not memoized

What it is

Dashboard.tsx:294–295: buildHistory(SIGNAL_HISTORY) and buildHistory(TPI_HISTORY) are called directly in the render function. buildHistory iterates over up to 43 entries and constructs a new array on every single render — including renders triggered by the BTC price polling every 30s.

Why it matters

The BTC price polling triggers a state update every 30s, which re-renders the Dashboard, which re-runs buildHistory twice. With 43 entries this is currently fast, but it will grow.

How to fix

Wrap with useMemo: const rawHistory = useMemo(() => buildHistory(SIGNAL_HISTORY), [activeSignal, isPremium]).

Warning W7 — App.tsx makes 2 Supabase queries on every route change for logged-in users

What it is

The resolveFlow effect in App.tsx runs on every location.pathname change. For a logged-in user navigating between pages, it fires 2 Supabase queries (survey_responses + profiles) every time. This happens even when navigating between /dashboard → /account → /methodology repeatedly.

Why it matters

Unnecessary latency and Supabase read quota usage on every navigation. The profile/survey data is unlikely to change between page navigations within the same session.

How to fix

Cache the isPremium and hasSurvey results in React context or a useRef and only re-query when the user object changes (not on every path change).

Warning W8 — Signal count discrepancy: CLAUDE.md says 42 RAW_SIGNALS, actual array has 43

What it is

CLAUDE.md states "RAW_SIGNALS (42 signals)" but the actual RAW_SIGNALS array in Dashboard.tsx contains 43 entries (Oct 2019 → Mar 2026). The landing page and subscription page both display "43 signals" which is correct — but the CLAUDE.md documentation is wrong.

Why it matters

Future Claude Code sessions (and human developers) reading CLAUDE.md will think there are 42 signals and may off-by-one when adding new ones.

How to fix

Update CLAUDE.md: "RAW_SIGNALS (43 signals)" and "TPI_SIGNALS (27 signals)".

Warning W9 — AccountPage uses re-sign-in to verify current password — triggers a full auth event

What it is

AccountPage.tsx:89: to verify the current password before changing it, the code calls supabase.auth.signInWithPassword(). This fires a new sign-in event which triggers onAuthStateChange in AuthContext and may cause unintended navigation via resolveFlow in App.tsx.

Why it matters

In practice this seems stable (the user is already logged in, so resolveFlow just confirms their state), but it's a fragile pattern that could cause a redirect loop if the timing is unlucky, and it creates an unnecessary auth event in Supabase logs.

How to fix

Supabase doesn't expose a "verify password without login" API, so the current approach is the best available option. Mitigate by adding a guard in resolveFlow that skips re-routing if location.pathname === '/account'.

🟢 Suggestions

Nice-to-have improvements — polish, future-proofing, conversion.

Suggestion S1 — No error boundary — a render crash takes down the entire app

There is no ErrorBoundary component anywhere. If any component throws during render (e.g. a malformed Supabase payload causing signal.price.toLocaleString() to crash), the entire app goes blank. Add a top-level ErrorBoundary in App.tsx with a friendly fallback UI.

Suggestion S2 — Supabase client has no error handling for missing env vars

src/lib/supabase.ts calls createClient(supabaseUrl, supabaseAnonKey) with no guard for undefined values. If VITE_SUPABASE_URL is missing in a deployment, the app fails silently with cryptic Supabase errors. Add: if (!supabaseUrl || !supabaseAnonKey) throw new Error('Missing Supabase environment variables').

Suggestion S3 — BTC price fetch has no CORS fallback if Binance blocks the request

The Binance API is fetched directly from the browser on a 30s interval. Binance can (and does) block requests from certain regions and network types. The catch block already handles failures gracefully by setting btcPrice(null), but the live P&L row then shows nothing with no explanation. Consider adding a "price unavailable" label when btcPrice is null after the first fetch attempt.

Suggestion S4 — signOut clears a localStorage key that is never written

AuthContext.tsx:120: localStorage.removeItem('tradinsight_flow'). Searching the codebase, tradinsight_flow is never written to localStorage. This is likely a vestigial cleanup from an earlier navigation state system that was replaced by surveyIntentional in App.tsx. Safe to remove.

Suggestion S5 — Mobile dashboard header hides username and plan type

On mobile (sm:hidden), the user's name and Free/Premium plan badge are hidden. For a paying premium user, not seeing their plan status on their primary screen is a missed trust signal. Consider showing a minimal plan badge (gold dot for premium) in the mobile header.

Suggestion S6 — Conversion: no upgrade CTA visible when free user has no active signal

When there is no active signal, the signal card shows an empty state ("No active signal. Market conditions not met."). For free users in this state, there is no upgrade prompt in the top section — they would have to scroll down to the signal history table to see the upgrade banner. Consider adding a subtle upgrade nudge to the empty signal state for free users.

Suggestion S7 — Landing page: "How it Works" link only shows for logged-in users

LandingPage.tsx:27: onMethodology && (...) — the "How it Works" nav link only renders if onMethodology is defined. In App.tsx:202, this prop is only passed when user exists. Logged-out visitors cannot navigate to the Methodology page from the landing page nav, even though it's a key trust-building page. Consider making it a direct <a href="/methodology"> or always passing the handler.
